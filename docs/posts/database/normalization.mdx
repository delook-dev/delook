# 정규화(Normalization)

## 1. 정규화란?

정규화는 관계형 데이터베이스에서 테이블을 논리적으로 구조화하는 설계 기법입니다. 체계적인 규칙에 따라 테이블을 분해하여 효율적이고 신뢰할 수 있는 데이터 구조를 만드는 과정으로, 현대 데이터베이스 설계의 핵심 원칙입니다.

## 2. 해결하려는 문제들

### 이상 현상(Anomaly)

- **삽입 이상**: 새로운 데이터 삽입 시 불필요한 정보까지 입력해야 하는 문제
- **갱신 이상**: 데이터 수정 시 여러 곳을 동시에 수정해야 하는 문제
- **삭제 이상**: 데이터 삭제 시 필요한 정보까지 함께 삭제되는 문제

### 데이터 품질 문제

- 같은 정보가 여러 곳에 중복 저장
- 중복으로 인한 불일치 발생 가능성
- 불필요한 저장 공간 사용

## 3. 정규화 단계별 과정

#### **학생수강정보테이블** (비정규화 상태 예시)

| 학번 | 이름   | 전공 | 수강과목                           | 교수명                 | 학점    |
| ---- | ------ | ---- | ---------------------------------- | ---------------------- | ------- |
| 1001 | 김철수 | 컴공 | 프로그래밍, 자료구조, 데이터베이스 | 박교수, 이교수, 김교수 | 3, 3, 2 |
| 1002 | 이영희 | 경영 | 경영학원론, 회계학                 | 최교수, 정교수         | 2, 3    |

**문제점**:

- 수강과목, 교수명, 학점 필드가 여러 값을 가집니다. (원자값 위반)
- 콤마로 구분된 다중값이 하나의 셀에 저장됩니다.

### 1) 제1정규화 (1NF) - 원자값 만족

각 속성이 원자값(더 이상 분해할 수 없는 값)을 가지도록 분해합니다.

#### **학생수강정보테이블**

| 학번 | 이름   | 전공 | 수강과목     | 교수명 | 학점 |
| ---- | ------ | ---- | ------------ | ------ | ---- |
| 1001 | 김철수 | 컴공 | 프로그래밍   | 박교수 | 3    |
| 1001 | 김철수 | 컴공 | 자료구조     | 이교수 | 3    |
| 1001 | 김철수 | 컴공 | 데이터베이스 | 김교수 | 2    |
| 1002 | 이영희 | 경영 | 경영학원론   | 최교수 | 2    |
| 1002 | 이영희 | 경영 | 회계학       | 정교수 | 3    |

**개선점**: 모든 필드가 원자값을 가집니다.  
**남은 문제점**: 학생 정보와 과목 정보가 계속 중복됩니다.

### 2) 제2정규화 (2NF) - 부분 함수 종속 제거

기본키가 아닌 속성이 기본키 전체에 종속되어야 합니다.

기본키: (학번, 수강과목)

- 이름, 전공은 학번에만 종속 → 부분 함수 종속 발생
- 교수명, 학점은 수강과목에만 종속 → 부분 함수 종속 발생

#### **학생테이블**

| 학번 | 이름   | 전공 |
| ---- | ------ | ---- |
| 1001 | 김철수 | 컴공 |
| 1002 | 이영희 | 경영 |
| 1003 | 박민수 | 컴공 |

#### **과목테이블**

| 수강과목     | 교수명 | 학점 |
| ------------ | ------ | ---- |
| 프로그래밍   | 박교수 | 3    |
| 자료구조     | 이교수 | 3    |
| 데이터베이스 | 김교수 | 2    |
| 경영학원론   | 최교수 | 2    |
| 회계학       | 정교수 | 3    |

#### **수강테이블**

| 학번 | 수강과목     |
| ---- | ------------ |
| 1001 | 프로그래밍   |
| 1001 | 자료구조     |
| 1001 | 데이터베이스 |
| 1002 | 경영학원론   |
| 1002 | 회계학       |
| 1003 | 프로그래밍   |

### 3) 제3정규화 (3NF) - 이행적 함수 종속 제거

기본키가 아닌 속성들 간의 종속성을 제거합니다.  
**이행적 종속:** 과목테이블에서 수강과목 → 교수명 → 소속학과

#### **학생테이블**

| 학번 | 이름   | 전공 |
| ---- | ------ | ---- |
| 1001 | 김철수 | 컴공 |
| 1002 | 이영희 | 경영 |
| 1003 | 박민수 | 컴공 |

#### **교수테이블**

| 교수ID | 교수명 | 소속학과 |
| ------ | ------ | -------- |
| P001   | 박교수 | 컴공     |
| P002   | 이교수 | 컴공     |
| P003   | 김교수 | 컴공     |
| P004   | 최교수 | 경영     |
| P005   | 정교수 | 경영     |

#### **과목테이블**

| 과목코드 | 수강과목     | 교수ID | 학점 |
| -------- | ------------ | ------ | ---- |
| CS101    | 프로그래밍   | P001   | 3    |
| CS102    | 자료구조     | P002   | 3    |
| CS201    | 데이터베이스 | P003   | 2    |
| BU101    | 경영학원론   | P004   | 2    |
| BU102    | 회계학       | P005   | 3    |

#### **수강테이블**

| 학번 | 과목코드 |
| ---- | -------- |
| 1001 | CS101    |
| 1001 | CS102    |
| 1001 | CS201    |
| 1002 | BU101    |
| 1002 | BU102    |
| 1003 | CS101    |

**최종 개선점**:

- 모든 데이터 중복 제거
- 각 테이블이 단일 주제에 대한 정보만 저장
- 데이터 일관성과 무결성 보장
- 이상 현상(삽입/수정/삭제 이상) 방지

## 4. 고차 정규화

### BCNF (보이스-코드 정규형)

제3정규형을 만족하면서 모든 결정자가 후보키여야 합니다.

- 3NF에서 발생할 수 있는 예외적인 이상 현상까지 제거합니다.
- 실무에서는 3NF까지 적용하는 것이 일반적입니다.

### 제4정규화 (4NF)

BCNF를 만족하면서 다중값 종속성을 제거합니다.

- 한 테이블 내에서 독립적인 다중값 속성들을 분리합니다.

### 제5정규화 (5NF)

제4정규형을 만족하면서 조인 종속성을 제거합니다.

- 테이블이 더 작은 테이블들로 분해되었다가 다시 조인될 때 정보 손실이 없어야 합니다.

---

## 5. 정규화의 단점

- **성능 저하**: 여러 테이블을 조인해야 하므로 쿼리가 복잡해집니다.
- **복잡성 증가**: 테이블 수가 증가하여 관리가 복잡해집니다.
- **조인 비용**: 조인 연산으로 인한 처리 시간이 증가합니다.
- **개발 복잡도**: 애플리케이션 로직이 복잡해져 개발 시간이 증가합니다.

```sql
-- 정규화 전: 간단한 쿼리
SELECT 학번, 이름, 수강과목, 교수명
FROM 학생수강정보테이블
WHERE 학번 = 1001;

-- 정규화 후: 복잡한 조인 쿼리
SELECT s.학번, s.이름, c.수강과목, p.교수명
FROM 학생테이블 s
JOIN 수강테이블 sc ON s.학번 = sc.학번
JOIN 과목테이블 c ON sc.과목코드 = c.과목코드
JOIN 교수테이블 p ON c.교수ID = p.교수ID
WHERE s.학번 = 1001;
```

## 6. 비정규화(Denormalization)

### 비정규화가 필요한 경우

- **성능 최적화**: 조회 성능이 중요한 시스템에서 조인 비용을 줄이기 위해
- **읽기 중심 시스템**: 데이터 변경보다 조회가 빈번한 시스템
- **데이터 웨어하우스**: 분석 성능 향상을 위한 스타 스키마 적용
- **실시간 시스템**: 응답 시간이 중요한 시스템에서 빠른 응답을 위해

### 비정규화 전략

- **수평 비정규화**: 자주 조인되는 테이블들을 합침
- **수직 비정규화**: 계산된 필드나 집계 필드를 추가
- **테이블 분할**: 성능을 위해 테이블을 용도별로 분리
- **중복 허용**: 조회 성능을 위해 의도적인 중복 데이터 저장

```
주문요약테이블 (비정규화 예시)
주문ID | 고객명 | 주문일자 | 총주문금액 | 주문상품수
1001 | 김철수 | 2025-09-01 | 150,000 | 3
1002 | 이영희 | 2025-09-02 | 85,000 | 2
1003 | 박민수 | 2025-09-03 | 220,000 | 5
```

## 7. 정규화 적용 지침

### 언제 정규화를 적용할까?

- **OLTP(Online Transaction Processing) 시스템**: 트랜잭션 처리가 중요한 시스템에서는 3NF까지 적용
- **데이터 무결성이 중요한 시스템**: 금융, 의료 등 정확성이 중요한 분야
- **데이터 입력/수정이 빈번한 시스템**: 이상 현상 방지가 중요

### 언제 비정규화를 고려할까?

- **OLAP(Online Analytical Processing) 시스템**: 분석과 리포팅이 주목적인 시스템
- **읽기 전용 시스템**: 데이터 변경이 거의 없는 시스템
- **성능이 중요한 시스템**: 응답 시간이 비즈니스에 직접적인 영향을 주는 경우

### 실무 적용 원칙

- 대부분의 시스템에서는 **제3정규형(3NF)까지 적용**하는 것이 일반적입니다.
- 성능 문제가 발생하는 부분에 대해서만 **선택적으로 비정규화** 적용합니다.
- **데이터 무결성과 성능 사이의 균형**을 고려하여 설계해야 합니다.
- 시스템의 특성(OLTP vs OLAP)을 고려한 설계 전략 수립이 중요합니다.

---

## 8. 결론

- 정규화는 데이터 무결성과 일관성을 보장하는 데이터베이스 설계의 핵심 원칙입니다.
- 제3정규형까지 적용하는 것이 일반적이며, 필요에 따라 선택적으로 비정규화를 적용합니다.
- 시스템의 특성과 요구사항을 고려하여 정규화 수준을 결정해야 합니다.
- 데이터 무결성과 성능 사이의 균형을 맞추는 것이 성공적인 데이터베이스 설계의 핵심입니다.
